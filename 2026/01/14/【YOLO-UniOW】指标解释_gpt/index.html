According to a document from 2024-12-30（YOLO-UniOW 论文），你在第 4 节 Experiments 里看到的指标，核心可以分成两大类：**开放词表检测（OVD, open-vocabulary detection）用的指标（表 1），以及开放世界检测（OWOD, open-world object detection）**用的指标（表 2/3/4）。下面我把论文里出现的指标逐个拆开讲清楚：它们怎么定义、评估模型什么能力、以及读表时最容易迷糊的点。

先打地基：IoU、TP/FP、Precision/Recall、PR 曲线是什么关系？

很多指标（AP/mAP/U-Recall/WI）本质都建立在同一套“检测评测”逻辑上：

  • IoU（Intersection over Union）：预测框与 GT 框（真实标注框）重叠程度 IoU=Area(pred∩gt)Area(pred∪gt) 通常会设一个阈值（如 0.5），IoU≥阈值才认为“框对上了”。
  • TP / FP / FN
    • TP：预测框匹配到某个 GT（IoU 达标），且类别也对（或在 unknown 评估里标签逻辑对）。
    • FP：预测框没匹配到任何 GT（或类别不对）。
    • FN：某个 GT 没被任何预测框检出。
  • Precision / Recall Precision=TPTP+FP,Recall=TPTP+FN
  • PR 曲线：把预测框按置信度从高到低扫一遍，阈值不断放宽时会得到一条 Precision-Recall 曲线。AP/mAP 就是在算这条曲线下的面积（或等价的离散近似）。

表 1 的指标（OVD / LVIS）：AP、APr、APc、APf、Params、FPS、FPSf

表 1 是 LVIS minival 的 zero-shot OVD 评测。它同时报告“效果指标”和“效率指标”。

1) AP（Average Precision）

是什么： AP 是“平均精度”，本质是某个评测协议下 PR 曲线的面积。 在 LVIS/COCO 风格里，AP 通常不是单一 IoU 的 AP，而是会在多个 IoU 阈值上取平均。

  • LVIS 论文明确说它采用 COCO-style AP：对类别与不同 IoU 阈值的表现做平均统计。
  • COCO 官方评测代码里常用的 IoU 阈值序列是 0.50 到 0.95，步长 0.05（也就是你经常看到的 AP@$$ .5:.95 $$）。

评估什么能力：

  • 在 OVD 场景里，AP 衡量的是：模型是否能正确定位 + 正确把框分到对应文本类别（zero-shot 泛化能力）。

常见误区：

  • 很多人以为 AP 就是 IoU=0.5 的 AP。但在 LVIS/COCO 主指标里，AP 往往是多 IoU 平均的（更严格）。

2) APr / APc / APf（LVIS 的 long-tail 分组 AP）

LVIS 把类别按训练集中出现频次分成三档，并分别计算 AP：

  • rare：该类别在训练集中出现图片数 1–10
  • common：11–100
  • frequent：>100

并且 LVIS 也明确说明下标 r/c/f 就是指 rare/common/frequent 子集上的 AP。

所以：

  • APr：只在 rare 类别集合上算 AP
  • APc：只在 common 类别集合上算 AP
  • APf：只在 frequent 类别集合上算 AP

评估什么能力：

  • 这是在测试模型对 长尾类别 的泛化：
    • APr 高说明模型对稀有/少样本类别的 zero-shot 能力更强（尤其重要）。
    • APf 往往更容易高，因为 frequent 类更“常见”。

3) Params

是什么： 参数量（通常以 M=百万为单位）。 评估什么能力： 严格说不是“准确性指标”，而是模型体量/存储/部署成本的衡量。Params 越小通常越利于边端部署，但也可能限制上限。

4) FPS 与 FPSf（速度指标）

表 1 里 YOLO-UniOW 明确解释了：

  • FPS：端到端推理速度（包含后处理，如 NMS），并且为了测 FPS，会在 NMS 前把预测框数量限制在 3000 以内。
  • FPSf：只算前向（forward）过程的速度，不包含后处理。

评估什么能力：

  • FPS更贴近真实部署吞吐（“跑起来每秒几帧”）。
  • FPSf更反映“网络本体有多快”，常用于对比不同后处理开销（比如 NMS-heavy vs NMS-free）。

5) 你在表 1 里看到的 “a/b” 不是新指标，而是两种 head 的 AP 并列写法

表 1 caption 写得很清楚：它把 **one-to-one head（左）**和 **one-to-many head（右）**的 AP 都报出来了。

表 2 的指标（OWOD：M-OWODB / S-OWODB）：U-Recall、PK/CK/Both mAP

表 2 是开放世界检测的经典评测：既要识别已知类，又要把未知物体“当作 unknown 检出来”。表头明确它比较的是 U-Recall 和 known classes 的 mAP。

6) mAP（mean Average Precision）在 OWOD 里通常用 mAP@0.5

在 OWOD 经典协议里，Joseph 等人的 OWOD 工作明确：用 IoU=0.5 的 mAP 来评估已知类（并说明与既有增量检测文献保持一致）。

PROB 这篇（也是 M-OWODB/S-OWODB 常用对比方法之一）在表格说明中也写得非常明确：它对比的是 known class 的 mAP@0.5，并且拆成 previously/current/both。

所以你会看到一个很关键的差异：

  • 表 1（LVIS OVD）：更像 COCO-style 的 AP@$$ .5:.95 $$（更严格）
  • 表 2（OWOD benchmark）：很多工作用 mAP@0.5（更接近 VOC-style）

7) PK / CK / Both（mAP 的三种拆分）

这是 OWOD/增量检测里非常关键的“读表逻辑”：

  • PK（Previously Known）mAP：在“以前学过的已知类”上算 mAP
  • CK（Currently Known）mAP：在“这一轮新引入的已知类”上算 mAP
  • Both mAP：在“截至目前所有已知类（PK∪CK）”上算 mAP

PROB 论文写明：在持续/增量学习评估里，mAP 会拆成 previously 和 newly introduced classes。 同时 OWOD 的任务设定里，“以前任务的类是 known、未来任务的类暂时视为 unknown”的逻辑也在原始 OWOD 协议里写得很清楚。

评估什么能力：

  • PK mAP：衡量“抗遗忘”（catastrophic forgetting）——新任务学了以后，旧类还能不能认？
  • CK mAP：衡量“学新类的能力”
  • Both mAP：整体已知类检测能力

读表技巧：

  • CK 高但 PK 掉很多 → 学新类可以，但遗忘严重
  • PK 稳、CK 也高 → 增量学习做得好
  • Both 只是总体，最好结合 PK/CK 一起看

8) U-Recall（Unknown Recall / Unknown class recall）

定义（最关键）： PROB 论文给了非常直接的定义： U-Recall（unknown object recall）是 “检测到的未知物体数 / 标注的未知物体总数”；并解释了为什么不用 unknown 的 mAP：因为并不是所有 unknown 都完整标注。

OW-DETR 的补充材料也强调：U-Recall 直接对应 unknown 类，衡量模型“检索/找回未知实例”的能力。

评估什么能力：

  • 这就是 OWOD 的“开世界感知能力”：模型能不能把没见过的物体从背景里“挖出来”，并标成 unknown。

非常常见的坑：

  • U-Recall 高不一定代表好，有些方法可能通过“更激进地报 unknown 框”来抬 recall，但会带来更多 FP，进而伤害已知类精度（这就是为什么还需要 WI/A-OSE）。

9) 为什么 Task 4 往往不报 U-Recall？

因为到最后一个任务时，很多协议会变成“所有类都已知”，没有 unknown GT 了，所以 U-Recall（以及一些 unknown 相关指标）不可计算。OW-DETR 与 PROB 都明确提到这一点。

表 3/4 的指标（OWOD 扩展）：WI、A-OSE（以及论文提到的 Recall）

YOLO-UniOW 在文中说明：为了评估检测 unknown 类的能力，他们用 Recall（也就是 U-Recall 这类思想）；另外还会报告 WI 和 A-OSE 来衡量 unknown 对 known 预测的干扰，但认为它们不够稳定，所以“参考为主”。

这两个指标来自 OWOD 经典定义（Joseph 等）：

10) WI（Wilderness Impact）

定义（带公式）：

WI=PKPK∪U−1

其中：

  • (P_K)：只在已知类上评估时的 precision
  • (P_{K\cup U})：把 unknown 也放进来一起评估时的 precision 并且是在固定 recall 水平 (R=0.8) 处测量（Joseph 的实验统一用 0.8）。

直觉：

  • unknown 加进来后，如果模型把 unknown 误当成 known，会造成 FP 增多 → precision 掉
  • precision 掉得越多，WI 越大
  • 所以 WI 越低越好（表格里通常标 (↓)）

评估什么能力：

  • “开放集干扰下的稳健性”：unknown 存在时，已知类检测的精度会被拖垮多少。

11) A-OSE（Absolute Open-Set Error）

定义： A-OSE 是一个“计数型”指标：unknown 物体被错误分类为任意 known 类的数量。 （越多说明越混淆）

评估什么能力：

  • 模型是否能把 unknown 与 known 分开（降低“unknown→known 的误分类”）。

一句话把所有指标和“模型能力”对齐

你可以用下面这张“能力-指标”对照表来建立直觉：

  • 开放词表/zero-shot 分类+定位能力（OVD）：AP（以及 APr/APc/APf）
    • 尤其看 APr：长尾、稀有类的泛化。
  • 开放世界未知物体发现能力（OWOD）：U-Recall
    • U-Recall = 找回多少 unknown。
  • unknown 不要干扰 known（开放集稳健性）：WI、A-OSE
    • WI 看 precision 掉多少；A-OSE 看 unknown 被错分成 known 的次数。
  • 增量学习与遗忘（Continual / Incremental）：PK/CK/Both mAP
    • PK = 旧知识保留；CK = 新知识学习；Both = 总体。
  • 效率与部署友好：Params、FPS、FPSf
    • FPSf 不含后处理；FPS 更贴近真实吞吐。

你接下来怎么“彻底搞懂”读表（给你一个实用检查清单）

当你看到一行结果时，按这个顺序读：

  1. 先看 mAP（Both）：已知类总体检测能力怎么样？
  2. 再看 PK vs CK：是否遗忘？是否学新类？
  3. 看 U-Recall：unknown 找回能力强不强？
  4. 若给了 WI/A-OSE：U-Recall 的提升是不是靠“乱报框/误报 known”换来的？（WI/A-OSE 会暴露这种情况）
  5. 看 FPS/FPSf + Params：性能提升是否值得部署成本？
