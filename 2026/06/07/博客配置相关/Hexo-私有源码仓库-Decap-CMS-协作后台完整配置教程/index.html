Hexo 私有源码仓库 + Decap CMS 协作后台完整配置教程

这篇文章是给未来的自己留的一份完整回溯手册，目标不是只记录“最后结果”，而是把从零开始搭起这一套博客协作后台的每一步都写清楚。后面如果我要重配一遍，或者想知道某个设置为什么这么做，直接按这篇操作即可。

最终我们做出来的是这样一套方案：

  • qieliqiean/blog-source：私有仓库，存放 Hexo 源码、文章源文件、后台配置
  • qieliqiean/blog：公开仓库，存放最终生成的静态页面
  • Decap CMS 后台直接编辑私有仓库 blog-source
  • GitHub Actions 在私有仓库里构建 Hexo，并把 public/ 推送到公开仓库 blog
  • 好友作为 blog-source 协作者，通过 GitHub 登录后台共同写文章

这套方案的核心优点是：

  • 源码、草稿、私密内容不暴露
  • 博客前台仍然可以公开访问
  • 协作者不需要本地配置 Hexo，也能直接在网页上发文
  • 本地写作和后台写作可以共存

1. 为什么选择这套双仓库方案

最开始最容易想到的做法，是把整个 Hexo 源码仓库直接公开，然后让后台直接写公开仓库。但这种做法有几个明显问题：

  • 源码、草稿、未整理内容、私密页面可能被看到
  • 如果有加密文章、备份文件、实验性草稿，都会跟着一起暴露
  • 朋友虽然能编辑，但实际上能看到整份博客源码

所以后面改成了更稳的方案：

  1. 源码仓库私有
  2. 生成后的静态页面仓库公开
  3. 后台只操作私有仓库
  4. 公共博客只接收编译结果

这就是本文的基础架构。

13. 本地 Hexo 写作和后台写作的关系

这部分一定要彻底搞清楚，否则后面很容易担心“会不会覆盖”。

13.1 CMS 写文章时到底发生了什么

CMS 保存或发布时，本质上是在做这件事：

  • 向私有仓库 blog-source 的 source 分支提交 Git commit

然后私有仓库的 Actions 再去：

  • 构建 Hexo
  • 把静态页面推到公开仓库 blog

13.2 本地仓库不会自动更新

你本地的 C:\blog 只是 blog-source 的一份克隆。

所以：

  • 后台新建文章后，本地不会自动出现
  • 必须手动 git pull 才会同步下来

13.3 hexo clean、hexo g、hexo s 不会帮你同步远端

这些命令只作用于本地当前文件：

  • hexo clean：清理缓存和 public
  • hexo g：根据本地文章生成静态页面
  • hexo s：启动本地预览服务

它们都不会自动把 CMS 新写的文章拉回本地。

13.4 正确理解“覆盖”问题

一般情况下，不会默默覆盖。

更准确地说：

  • 如果远端比你新，你直接 git push 往往会被拒绝
  • Git 会要求你先同步远端
  • 如果你和后台改的是不同文件，通常能自动合并
  • 如果改的是同一篇文章，才可能发生冲突，需要手动处理

最不该做的事情是：

除非你非常确定自己在做什么，否则不要对这套协作仓库强推。

  • 本地预览用 hexo clean、hexo g、hexo s

  • 正式发布统一用 git push private-source source

这里还有一个非常重要的习惯变化：

  • 以后不要再把 hexo deploy 或 hexo d 当作正式发布方式

原因是现在公开仓库 qieliqiean/blog 已经只是静态页面产物仓库，而不是源码协作入口。真正的源码入口是私有仓库 qieliqiean/blog-source。

17. 这套方案目前的结论

到现在为止，这套方案已经证明是可行的：

  • 博客前台能正常访问
  • CMS 后台能正常登录
  • 协作者能进入后台
  • 后台发布可以触发 Actions
  • 公开博客能自动更新

而且这套结构也给后续继续优化后台留了空间，比如：

  • 进一步简化字段
  • 优化分类和标签输入体验
  • 改善封面图和媒体上传流程
  • 继续调整 快速发布 与 文章库 的交互
