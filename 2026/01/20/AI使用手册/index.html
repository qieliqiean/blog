Codex 命令精简速查表

CLI 终端命令

以下列出 Codex CLI 在终端中常用的命令及其用途，每条附有用途说明和示例：

命令 简要说明 示例
codex (无子命令) 启动交互式 Codex 会话（可选地直接附带提示）。启动后可在终端与 Codex 聊天，Codex 将读取当前目录内容并生成方案。 codex "解释这个代码库" – 在终端快速询问一个问题，Codex 读取项目并直接给出答案。
codex run "<指令>" 以一次性方式执行给定指令（可选地附带文件上下文），直接在终端输出结果。适合无需持续对话的单次请求。 codex run "总结此文件内容" --file README.md – 附带文件进行总结，让 Codex 读取 README.md 并输出概要。
codex edit <文件> "<说明>" 让 Codex 对指定文件进行修改。Codex 将读取文件内容，依据说明生成改动方案并直接编辑文件（需相应权限）。适用于让 AI 优化或修复代码片段。 codex edit src/Login.tsx "添加表单验证" – 让 Codex 在 Login.tsx 文件中添加表单验证逻辑。
codex resume [<ID> 或 --last] 恢复之前的对话会话，延续先前上下文继续工作。不加参数将列出最近会话供选择，--last 直接打开上次会话，亦可指定会话 ID 精确恢复。 codex resume – 列出最近会话并选择恢复 codex resume --last – 恢复最新一次会话。
codex login [--token <密钥>] 登陆 Codex 开发者账户。首次使用需通过浏览器 OAuth 或提供 API 密钥完成认证。--token 选项可使用 API Key 登录，省去浏览器流程。登录成功后可使用 codex whoami 查看当前账户信息。 codex login --token sk-... – 使用 OpenAI API 密钥登录。不带参数运行 codex login 则会启动浏览器进行 ChatGPT 账号认证。
codex exec "<指令>" 非交互模式执行 Codex 指令，以脚本方式运行并将结果打印到标准输出。可用于将 Codex 集成到自动化流程中。支持附加 --json 输出机器可读结果。 codex exec "修复 CI 中的失败测试" – 非交互地让 Codex 执行修复任务，完成后直接输出修复结果。
codex history 查看近期 Codex 执行历史或会话列表，便于回顾先前生成的内容和操作记录。 codex history – 在终端列出最近使用 Codex 生成的历史记录，包含每次会话或指令的摘要。

VS Code 插件典型功能

在 VS Code 中安装 Codex 插件后，可通过以下功能与 Codex 交互，实现聊天、自动补全等集成开发体验：

功能/界面操作 简要说明 示例/触发方式
Codex 聊天面板 在 VS Code 侧边栏打开 Codex 聊天窗口，与 AI 以对话方式互动。支持从当前打开的文件获取上下文，使回答更贴合代码环境。用户可以在此输入需求，实时查看 Codex 回复并预览代码更改。 VS Code 命令面板中选择“Open Codex Panel”或使用已绑定快捷键打开聊天面板。打开后即可输入问题，如 “如何优化此函数？” Codex 将结合当前项目给出建议。
内联代码补全 Codex 在代码编辑器中提供类似 Copilot 的自动补全建议，帮助开发者以实时方式编写代码。根据已输入的上下文（如函数签名或注释），Codex 会实时预测后续代码。开发者可按 Tab 或相应快捷键接受建议，加速编码过程。 在代码中键入注释 “// TODO: 验证输入” 后暂停，Codex 可能会自动弹出建议代码。例如它可能补全一段输入验证逻辑，按 Tab 插入该片段。
文件/选区引用 将当前编辑的文件或选中代码片段添加到 Codex 上下文，便于 AI 理解问题所涉及的代码。这可以通过在聊天提示中使用 @文件名 引用，或使用 VS Code 提供的命令将选区/文件添加到对话线程来实现。 1）在对话中输入:请参考 @utils.py 实现上述功能，Codex 将读取 utils.py 内容再回答。 2）选中代码片段后，在命令面板选择“Codex: Add Selection to Thread”，将该片段作为上下文发给 Codex。
一键代码操作 通过 VS Code 插件的右键菜单或命令面板，触发特定的 Codex 自动化编码任务。插件内置了常用操作，如“Fix Selected Code”（自动修复选中的代码问题）、“Generate Unit Test”（为当前代码生成单元测试）等，将手动步骤交给 AI 完成。 在代码中选中一段存在问题的函数，实现以下操作：右键菜单选择“Codex: Fix Selected Code”，Codex 将理解选中内容并直接给出修改建议，然后在终端面板显示 diff 供用户审查并应用。

Slash 指令速查（CLI 会话中）

在 Codex 交互式会话中，可使用“/”触发内置指令，以快速控制会话行为或获取特定功能。下表按名称列出常用 Slash 指令及适用场景：

指令 说明 适用场景
/model 切换当前对话使用的模型及推理深度。可选较快的通用模型或更深度的推理模型（如 gpt-4.1）。 需要在速度和推理能力间调整，例如复杂任务前切换更强模型。
/approvals 设置 Codex 操作的授权模式（Auto/Read-only/Full）。可动态放宽或收紧 Codex 执行修改和运行代码的权限。 根据当前信任程度调整，让 Codex 在安全模式下提建议或自动应用改动。
/new 开启一个新的对话线程。在不退出 CLI 的情况下重置对话上下文，开始处理一个全新问题。 需要在同一项目中就不同任务与 Codex 重新对话时使用。
/init 生成初始化的 AGENTS.md 指南文件。该文件用于向 Codex 提供项目背景、风格指南和约束。 初次在新项目目录使用 Codex 时，快速创建指导文件帮助 AI 理解项目规范。
/compact 压缩对话上下文。Codex 会总结当前可见的对话内容，从而释放 token 容量以继续长对话。 当对话轮数很多接近模型上下文长度限制时，使用此命令缓解上下文超长问题。
/diff 显示 Codex 所做的代码改动差异（Git diff）。包括未被 Git 追踪的文件变更。 在允许 Codex 编辑代码后查看具体改动，再决定是否接受更改。
/mention 将指定文件或目录附加到对话上下文中。使 Codex 接下来可以访问该文件内容进行分析。 需要 Codex 关注某个文件（如日志或源码）时使用，在提问前先 /mention 路径 引入文件内容。
/status 显示当前会话的状态信息：包括所用模型、权限模式、可写目录列表以及剩余上下文窗口大小等。 确认当前配置或检查 token 消耗情况时使用，确保后续操作有足够上下文容量。
/resume 从会话列表中恢复先前保存的对话。可继续之前未完成的任务，对话历史和上下文都会保留。 需延续昨日讨论的话题或任务时使用，避免从头提供背景。
/fork 将当前或指定的历史对话分叉为新线程。保留原对话记录的同时开启新分支，以便尝试不同的解决思路。 面临一个问题的多种解法时，可以 fork 会话分别探索，而不会丢失原方案。
/review 让 Codex 审查当前工作区的代码改动并给出反馈。可选择不同范围（如对比主分支、未提交更改、指定 commit）进行代码审阅。 提交代码前希望静态检查潜在问题时使用，例如运行 /review uncommitted 查看所有尚未提交改动的风险提示。
/quit (/exit) 退出 Codex CLI 会话。两者作用相同。 完成所有任务后退出交互环境时使用。

特定场景下常用命令推荐

代码修改场景

当利用 Codex 进行代码重构、调试修复等任务时，可组合使用以下命令以提高效率：

命令/功能 推荐用法 示例
Read-Only 模式 首次让 Codex 提出代码修改建议时，建议将授权模式设为只读（/approvals 选择 Read Only）。此模式下 Codex 只能提出修改方案而不会直接改动代码，方便开发者先行审查。 触发 CodeX 重构函数时先保持 Read-Only 模式，观察 Codex 给出的修改计划是否合理，再决定是否应用。
/diff 查看改动 在 Codex 给出修改并申请执行后，使用 /diff 查看具体的代码差异。通过 diff 可以直观了解 Codex 将进行的改动内容，确保符合预期再批准。 Codex: “我已完成修改，是否应用更改？” 用户: 输入 /diff 查看具体代码改动，再决定是否让 Codex 执行。
切换 Auto 模式 如果对 Codex 建议的改动已审核通过，可通过 /approvals 将模式切换为 Auto，让 Codex 自动应用修改。Auto 模式允许 Codex 在工作区内直接编辑文件和运行命令。 在确认重构方案可行后，切换到 Auto 模式，Codex 将自动执行代码改动并运行相关构建或测试以验证结果。
/review 代码审查 在完成一轮修改后，使用 /review 让 Codex 对当前项目改动进行审查。Codex 会检查代码质量、潜在错误和改进点，并反馈总结报告。 完成修复后运行 /review，Codex 可能指出变量命名不一致或潜在边缘情况遗漏等问题，帮助进一步完善代码。
/mention 聚焦文件 在让 Codex 审查或修改时，可用 /mention 文件路径 将特定文件纳入上下文焦点。这可令 Codex 将注意力限定在相关文件，避免干扰其他无关部分。 在调试特定模块时，先 /mention src/utils.js，接着提问 “请检查此文件中的潜在问题”，Codex 将只分析 utils.js 内容。
分叉会话 当对代码修改有多种方案犹豫不决时，可使用 /fork 将当前对话复制为新线程进行平行探索。这样可以分别查看不同方案的效果，而原始对话内容仍保留备查。 在尝试不同算法优化思路时，先 fork 会话A为会话B，在B中让 Codex 实现替代方案，同时保留A以比较两种方案性能。

论文分析场景

当借助 Codex 阅读和分析学术论文或长文档时，建议使用以下命令组合以更好地处理长文本：

命令/功能 推荐用法 示例
文件内容导入 将论文文本导入 Codex 上下文：若论文以文本文件形式存在，可在对话开始时使用 /mention 文件路径 附加全文，使 Codex 能阅读全文内容；或者直接使用 CLI 命令附带文件，比如 codex run "总结这篇论文" --file paper.txt。 使用 codex run "请总结这篇论文的结论" --file paper.txt，Codex 将读取 paper.txt 并输出中文摘要，方便快速了解论文结论。
/model 切换模型 针对复杂的学术分析场景，可使用 /model 切换到更强大的模型（例如 GPT-4 系列）。高级模型具备更长的上下文和更强的推理能力，适合深入理解长篇论文和复杂内容。 在提问 “这篇论文的方法论有哪些局限？” 前，先 /model gpt-4.1 切换模型，以获得更详尽严谨的分析回答。
/compact 压缩上下文 若对话内容变得冗长且接近模型上下文长度限制，使用 /compact 压缩上下文。Codex 会总结当前讨论要点，以释放空间继续深入讨论。 与 Codex 连续讨论论文细节十余轮后对话变长，这时输入 /compact，Codex 将整理要点，使后续交流不受上下文长度限制。
分支讨论 面对论文的不同部分或观点，可通过 /fork 分支出新的对话线程分别讨论。这有助于分别深入研究不同章节内容，而不会把多主题混杂在一个对话中。 分析论文时，可将关于“实验结果解读”的提问与“理论推导细节”的提问放在两个 fork 出的会话中分别进行，使讨论更有条理。
/status 查看状态 在长文档分析过程中，随时使用 /status 查看当前模型、已用 token 数等状态。特别是大论文可能耗费大量上下文，检查 token 用量有助于决定何时需要压缩或终止对话。 输入 /status 可看到例如“剩余上下文容量：约 2000 tokens”等信息，提醒用户及时采取 /compact 等措施确保分析顺利进行。

OpenCode

OpenCode CLI 常用命令

命令/操作 说明 示例/用法
启动 OpenCode CLI 进入项目目录后运行 opencode 启动交互界面。可使用 CLI 执行与模型对话、代码分析和生成等操作。 cd myproject/ && opencode 启动 OpenCode，到终端 TUI 界面开始对话。
/init 初始化项目，扫描代码生成项目说明文件 AGENTS.md。该文件总结项目结构和约定，有助于 AI 理解代码（建议纳入版本管理）。 启动 OpenCode 后输入 /init 让 AI 分析当前仓库，生成 AGENTS.md 描述各模块职责。
模式切换 (Plan/Build) Plan 模式为只读分析模式，不自动改代码；Build 模式为可写模式，允许编辑文件和执行命令。按下 Tab 键可在两种模式之间切换。通常阅读代码用 Plan 模式，确认方案后再切换回 Build 模式实施修改。 例如，在 Plan 模式下让 AI 审阅代码不会修改文件；确认修改思路后按 Tab 切换到 Build 模式执行更改。
文件引用 (@) 在对话中引用项目文件或代码片段，提供完整上下文供 AI 理解。语法：@相对路径#L开始-L结束 可以引用特定行范围。引用越充分，AI 解答越准确。 询问函数作用时，可输入例如：请解释 @src/utils/parse.py#L10-L30 的逻辑，让 AI 基于引用代码给出解释。
撤销/重做修改 使用 /undo 撤销上一次由 AI 执行的代码改动，恢复修改前状态；使用 /redo 重做刚撤销的改动。方便在 AI 修复代码出错时回退或重apply更改。 在 Build 模式应用修复后发现问题，输入 /undo 撤销该改动；若误撤销则 /redo 恢复之。

OpenCode VS Code 插件功能

功能/操作 说明与用法 快捷键/触发
终端集成与快速启动 在 VS Code 内打开或聚焦一个 OpenCode 终端会话，在编辑器中直接使用 OpenCode。 打开/聚焦终端：Mac 下按 Cmd+Esc，Windows/Linux 下按 Ctrl+Esc；新建终端会话：按 Cmd+Shift+Esc（Win/Linux Ctrl+Shift+Esc） 强制创建新会话。
文件引用与上下文嵌入 插件可根据当前编辑器状态自动生成 @文件路径#行号 引用，提供代码上下文给 OpenCode。 选中代码后使用引用命令，会插入带行号范围的文件引用标记，AI 将基于引用内容理解问题。 在编辑器中选中一段代码，按 Cmd+Option+K（Win/Linux 下 Alt+Ctrl+K）即可插入当前文件及选区的引用标记到对话输入。
智能聊天面板 插件在侧边栏提供 OpenCode 聊天面板，可直接与 AI 对话。 与终端界面同步，支持上下文持续和多轮对话。适合在编写代码时随时提问 AI 或让 AI 解释选中的代码。 在 VS Code 侧边栏打开 OpenCode 面板，输入自然语言提问或命令，即可收到 AI 对答。（也可通过 VS Code 命令面板搜索 OpenCode 命令打开面板）。
代码解释与重构 支持右键菜单调用 OpenCode 对所选代码片段进行操作。如“解释此段代码”、“重构/优化此代码”等。 插件会将选中代码发送给 AI 并在聊天面板显示解释或重构建议，帮助理解代码意图或改进代码实现。 代码选择后右键，选择 OpenCode 提供的操作（如“Explain with OpenCode”或“Refactor with OpenCode”），即可在聊天面板查看 AI 给出的讲解或重构方案。
语义导航与问题定位 利用 AI 对项目的理解进行代码导航。可通过提问让 AI 查找某功能或变量的定义、引用位置，辅助定位问题根因。插件结合 OpenCode 的搜索工具（如 glob/grep）自动扫描项目代码，实现“按语义跳转”。 例如，在聊天中询问“函数calculateTotal在哪里定义？”插件将触发 AI 的文件搜索工具定位定义位置；再询问调用处，AI 可列出相关引用，点击结果即可在编辑器中打开对应文件行。

Slash 指令速查（交互式命令）

指令 作用 示例
/init 分析当前项目并生成 AGENTS.md 项目说明文件。帮助 AI 理解工程结构、模块职责和编码规范。 /init – 初始化项目后，自动打开生成的 AGENTS.md 查看项目结构概要。
/connect 连接模型提供商，配置 API Key 等凭据。执行命令后可选择 Anthropic、OpenAI 等模型并输入密钥，完成后续对话所需的模型授权。 /connect – 选择 OpenAI 并粘贴 API Key，配置完成后即可调用对应模型回答问题。
/models 列出可用的模型列表，或快速切换当前对话使用的 AI 模型。方便针对不同任务选择速度或精度合适的模型。 /models – 打开模型选择列表，切换为 claude-4-5 模型以提高代码理解能力。
/undo 撤销上一次 AI 执行的文件修改操作。会将受影响文件恢复到修改前的状态，便于快速回退错误修改。 /undo – 如果 AI 修复导致新错误，立即撤销该更改恢复原代码。
/redo 重做最近一次被撤销的修改。用于还原刚才撤销的更改（如果发现撤销不当），无需重新让 AI 生成。 /redo – 不小心撤销了正确修改？用此命令重做撤销，恢复 AI 提交的更改。
/share 生成当前对话的分享链接或文件，以便将对话内容发送给他人查看。分享记录包括用户提问和 AI 答复，便于协作调试。 /share – 获得一个会话URL，将其发送给同事共同审阅 AI 给出的代码修改建议。
/reset 重置会话，将对话上下文清空并开始一个全新会话。相当于打开了新的对话窗口，常用于长对话后的上下文清理或快速重新开始。 /reset – 当前话题偏离太多时，重置会话再重新提问，以免早前无关上下文干扰新回答。
/clear 清除当前对话的消息历史，但保留已加载的文件/项目上下文。适用于希望在不重新连接文件的情况下，让对话从空白对话记录开始。 /clear – 清空聊天记录后继续提问，让 AI 忽略之前的对话，只基于已有代码上下文回答。
/focus 专注模式：聚焦特定文件或范围的上下文，排除无关内容干扰。通过限制 AI 所考虑的上下文范围，提升大项目中某模块分析的准确性。
(注：功能类似通过只引用相关文件来“聚焦”上下文，实际可结合 /drop 等命令实现)		 输入 /focus src/module/，让 AI 仅关注 src/module/ 目录下代码进行分析。（需确保未聚焦部分已通过 /drop 移出上下文。）
/context 上下文检查：列出当前对话中纳入的文件、内容片段及其令牌占用情况，帮助用户了解会话上下文规模。可用于排查某文件是否已加载或判断是否需要精简上下文。 /context – 查看当前会话包含的文件列表和各自 token 用量，判断是否需要使用 /drop 移除某些文件以减小上下文。
