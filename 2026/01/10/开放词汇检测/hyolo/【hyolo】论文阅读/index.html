hYOLO Model: Enhancing Object Classification with Hierarchical Context in YOLOv8

查看原始论文 PDF（点击展开预览）

新窗口打开 PDF

Abstract

本节核心结论：论文在 YOLOv8 的分类分支里显式注入“类别层级结构”，让模型不仅预测最终细类，还能在多个层级上输出分类，并用层级一致性损失 + 层级指标来训练/评估，从而把“错得近 vs 错得远”的差别纳入学习目标。

摘要精读

  1. 现有问题：flat classification 不看层级

  • 传统 CNN/YOLO 的分类通常是扁平分类：直接在所有类别里选一个（比如 COCO 80 类就是 80-way）。

  • 但现实世界类别天然是有层级的（“饮料→牛奶→低脂牛奶”），忽略层级会导致：

    • 上下文理解差（类间关系没被用到）
    • 错误严重性不可控（把“2% 牛奶”错成“全脂牛奶” vs 错成“红酒”，损失一样离谱）
  1. 论文要做的事：端到端的层级检测+分类（基于 YOLO 系列）

  • 提出 hYOLO：在 YOLO 框架上做端到端end-to-end的检测与层级分类。

  • 贡献被摘要明确拆成三块（你后面复现也要按这三块落地）：

    1. 新的层级架构（hierarchical architecture）：多层级分类输出 + 层级信息融合
    2. 改过的 loss（modified loss）：按层级严重性加罚（hierarchy-aware penalty）
    3. 层级指标（performance metric）：评估时对“部分正确”给分（比如同一子树内错分给较高分）
  1. 实验设置：同一数据集，两种层级方式

  • 他们用“同一个数据集”的两种层级划分来验证观点：

    • Version 1：按“超市品类/系统分类”构树（不管视觉像不像）
    • Version 2：按视觉相似性来构层级（特别强调形状等视觉属性）

  • 摘要给出的结论倾向是：考虑视觉相似性的层级结构，更能体现优势。

关键概念/符号表

  1. hYOLO：本文提出的层级目标检测与分类模型（基于 YOLOv8 改造）。
  2. hierarchy depth l：层级深度（有多少个分类层级）。
  3. C_t / C_p：真实类别 / 预测类别（用于层级指标）。
  4. Ancest(C)：节点 C 的祖先集合（不含 root），用于层级 Precision/Recall。
  5. PrecHier, RecHier, Fβ,Hier：层级版 precision/recall/Fβ。
  6. α：层级一致性惩罚项的正则系数（越大越“强制遵守层级”）。
  7. δ_il：指示函数：类 i 是否是上一层父类的子类（是=1，否则=0）。
  8. conf_il：第 l 层对类 i 的置信度（用于“错得越自信罚得越重”）。

PyTorch 架构映射

摘要里的想法如何落到 YOLOv8/Ultralytics 代码

A) “层级架构”在代码里对应什么？

对应点：改 Detect Head 的分类分支

  • YOLOv8 检测头本质输出：(bbox 回归相关) + (cls logits)

  • hYOLO 做法：bbox 分支基本不动，重点改 cls branch：

    • 让模型在每个层级都输出一次分类（Level 0, Level 1, …, Level l-1）
    • 且把上层的特征/输出（论文里有不同 V1~V6 融合方案）concat 到下一层，形成“从粗到细”的上下文传递。

你在 Ultralytics 代码里最终会看到类似这种张量流（先给你一个“复现时要对齐的形状直觉”）：

  • backbone/neck 输出给 head：x = [P3, P4, P5]，每个是 (B, C, H, W)
  • 每个 level 的 cls 分支都会产生 logits：logits_l 形状近似 (B, n_cls_l, H, W)（每个尺度各一份，YOLO 是多尺度检测）
  • 融合通常就是 torch.cat([feat_current, feat_prev_level], dim=1) 后再卷积对齐通道。

B) “改过的 loss”在训练循环里对应什么？

  • 原 YOLOv8：一个 L_cls（BCE）

  • hYOLO：每个层级都有一套 loss（回归+分类），最后对层级求平均：

    • L = (1/l) * Σ_l (L_reg_l + L_cls_l)

  • 且对 l>0 的层级，在 L_cls_l 里加“非子类惩罚项”（由 α 控制）。

C) “层级指标”对应什么？

  • 训练/评估日志里不只看 flat 的 precision/recall/F1，而是按祖先交集计算 PrecHier/RecHier/Fβ,Hier，用来区分“错得近/远”。

Ⅰ Introduction

Introduction 这一节主要是在回答三件事：什么是层级分类、它为什么比 flat 更合理、以及为什么 YOLO 圈子里这事一直没被系统化做好。

1.1 本节核心结论

作者用“真实世界类别天然成树”的事实，论证 flat 分类在错误严重性、可解释性与大规模类别学习效率上先天吃亏，并指出层级数据/指标/loss 的缺失导致实践中仍偏向 flat；因此他们提出在 YOLOv8 上做端到端层级分类来补齐这整套闭环。

1.2 关键概念/符号表

  1. Hierarchical classification（层级分类）：类别按多层抽象组织成 taxonomy/树，模型可“从粗到细”逐层细化预测。
  2. Flat classification（扁平分类）：直接在最细粒度类别上分类，不利用层级关系（文中用生物分类法作对比）。
  3. Taxonomy / hierarchy levels（层级结构/层级数）：从 broad 到 specific 的多层级抽象（kingdom→…→species 例子）。
  4. Error control（错误可控）：希望错也“错得近”，避免跨大类的灾难性误判（文中以医疗风险分组举例）。
  5. More efficient learning（更高效学习）：粗到细的逐级学习，先学高层特征再学细粒度区分，提高收敛与泛化（文中继续沿用医疗示例展开）。
  6. Interpretability（可解释性）：预测路径可追踪（从大类到小类），更符合行业决策流程。
  7. Scalability（可扩展性）：把超大类别集合分解成多个更小的子问题，降低“大平面多分类”的难度与成本。
  8. Why flat still dominates（为何仍主流是 flat）：层级数据稀缺、构建难、且需要专门的层级指标与层级 loss 才能训练/评估好。
  9. End-to-end hierarchical model on YOLO（论文落点）：作者明确说要在 YOLO 家族上做端到端层级分类，并用“grocery store item classification”作为核心应用场景。

1.3 Introduction 精读

  1. 先把层级分类定义清楚（用生物分类法做类比）

  • flat：直接分到 species，不管 kingdom/phylum/class…

  • hierarchical：先预测大类，再逐层细化到最细类，让预测发生在“相关类的上下文”里。

    1. 层级分类的三大工程价值（这三点基本就是你复现报告的“动机段落”）

  • 错误可控：通过“保持在同一层级大类”降低灾难性误判概率（高风险领域更重要）。

  • 学习更高效：从粗到细的学习顺序，先学高层稳定模式，再做细粒度区分，有利于收敛与泛化。

  • 可解释 & 可扩展：能追踪决策路径；且把大规模分类拆成多个子问题，更可规模化。

    1. 为什么大家还是在用 flat（这段是“研究空白/痛点”）

  • flat 需要同时区分所有类、计算压力大；也无法体现“错得远更严重”。

  • 更关键：层级数据难构建 + 没有成熟的层级指标与层级 loss，训练/评估门槛高，所以 flat 仍占主流。

    1. 作者的承诺（引出后面 Methods）

  • 他们直接说：要做一个 基于 YOLO 家族的端到端层级分类模型，并用真实的“超市商品识别”来证明层级建模对 flat 的补强。

PyTorch 架构映射：Introduction 的观点在你代码里会落到哪些“硬改动”

把引言的三条价值直接映射到你复现时必须实现的三块（你后面看 Methods 会一一对应）：

  1. 错误可控（错得近） → hierarchy-aware loss / consistency penalty
    • 引言强调“错得远更糟”，这通常会落成：对跨父类/跨子树的预测加额外惩罚（你代码里就是 alpha 那类项的来源）。
  2. 更高效学习（粗到细） → 多层级分类输出 + coarse-to-fine 信息传递
    • 引言描述的是逐级细化的推理过程，因此实现上通常就是：head 里每个层级都有 cls logits，并把上层“粗类别信息”注入到下层（后面 V1~V6 架构就是在回答“怎么注入最有效”）。
  3. 可解释性 → 输出可追踪的层级路径 + 层级评估指标
    • 引言说“能追踪抽象层级”，工程上就意味着：你不仅要输出 leaf 类，还要能输出/评估每层的预测质量；并且需要配套层级指标来衡量“部分正确”。

2.1 原文过一遍

(A) 先总述：层级结构进深度学习，做法大体分两类

作者开头先说：把“类别层级结构（taxonomy）”引入深度学习，已经在 CV、文本分类等领域出现很多方法。总体上，大家主要从两条路线推进：

  • 改网络结构（architecture-level）：让网络在结构上“长得像一棵树”或显式输出多个层级。
  • 改训练目标（loss-level）：让损失函数体现“错得远 vs 错得近”的代价差异，并鼓励层级一致性。

这一段的作用：告诉你“后人都怎么做过”，为 hYOLO 的两大核心改动（结构 + loss）铺路。

(B) 结构改法：典型是“多输出/多分支 + 层级逐级细化”

作者在结构类工作里重点举了几种经典套路（你会发现它们的共同点都在多层级输出）：

  1. Branch CNN / 多分支输出（每个层级一个输出头） 思想是：同一张图，网络同时输出 level-0、level-1… 每层的分类结果。这样高层（粗类）能给低层（细类）提供上下文约束。
  2. HierarchyNet（更强的层级一致性 + 参数效率） 在多分支的基础上，强调更“层级化”的表示方式，让不同层级之间更一致、同时减少参数/提升一致性。
  3. Tree-CNN 这类“可扩展树结构” 重点是：层级树结构可以动态扩展（新增类别时不用从头训），适合大规模分类不断增长的场景。
  4. 还有一类替代思路：特征空间/度量学习 不一定把网络结构做成树，而是用 clustering / metric learning 的方式，让同一父类下的子类在 embedding 空间更“近”，减少类内方差、体现层级相似性。

这一段的作用：告诉你“结构上怎么显式编码层级”。hYOLO 后面提出 V1~V6，本质就是在 YOLOv8 head 上回答同一个问题：上层信息怎么注入下层最有效？

(C) Loss 改法：从“层级距离惩罚”到“软目标/课程学习”

作者接着总结 loss 相关工作，核心是：把层级信息放进 loss，让模型学会‘错得近一点’。大概有几类：

  1. Hierarchical loss / 语义层级损失 通过层级结构定义一种损失，让“可接受的近邻错误”罚得轻，“跨大类错误”罚得重。
  2. Severity-weighted（错误严重性加权） 显式按层级距离或分支距离给惩罚权重：错到更远的分支，loss 更大。
  3. Curriculum learning（课程学习） 用层级结构安排训练顺序：先学粗粒度，再学细粒度，让优化更稳定、更符合认知过程。
  4. Soft target regularization（软目标/软标签） 不把真值当 one-hot，而是给“同层/同父类”的类别分配一点概率质量，缓解长尾、提升泛化。

这一段的作用：告诉你“训练目标怎么编码层级”。hYOLO 后面那个 α 控制的层级一致性惩罚项，就是沿着这条线走的，但它把这件事落在 YOLOv8 检测训练里。

(D) YOLO 相关层级工作：重点提 YOLO9000 + “后来 YOLO 很强但层级没跟上”

原文中跟 YOLO 直接相关的叙事大概是：

  1. YOLO9000 / WordTree 是早期把层级概念带进 YOLO 的代表（能支持非常多类别）。 但作者指出它的局限：层级依赖固定的（并且不一定符合你任务的）粗粒度层级结构，而且整体方法在今天的“现代 YOLO 体系”里不够灵活/不够系统。
  2. 后面也有人做应用定制版的层级 YOLO（比如某些特定行业/场景），会改 NMS、改 detection layer、让模型能用层级信息。 但这些多是“点状应用”，缺少一个能泛化到现代 YOLO（尤其 YOLOv8 这种）的统一框架。
  3. 作者还快速回顾了 YOLOv3→v5→v6/v7→v8 的演进（多尺度、动态分配、解耦头等），最后落脚： YOLO 在检测速度/精度上突飞猛进，但层级分类这块仍缺“通用化、端到端、配套 loss + metric”的系统方案。

这一段的作用：把研究空白钉死——“不是没人做过层级 YOLO，但要么旧、要么太定制；而现代 YOLOv8 缺一个完整闭环”。

(E) 指标与评估：层级分类不能只看 flat accuracy/F1

最后作者提到：层级分类的评估不能只用 flat 指标，因为 flat 指标不会奖励“部分正确”。因此有人提出：

  • 基于祖先集合的层级 precision/recall/F：预测类与真值类的祖先集合交并来算，能区分不同类型的错。
  • 基于层级距离/成本的度量：预测错到多远，按距离给成本。

作者还明确说：他们在 YOLO 的实现里采用了这种层级评估思路，用来比较不同层级架构的效果。

这一段的作用：为论文后面提出的 hierarchical metric 做“合理性证明”，也提醒你复现时必须补齐评估代码，否则你看不出“错得近”的收益。

2.2按复现节奏重新归类

下面我用“复现工程视角”把 Related Work 里提到的东西分成三类，并告诉你 hYOLO 大概站在哪个位置。

A. 架构改法（architecture-level）

怎么把层级塞进网络输出/特征流？

主流套路 1：多层级多头输出（multi-head / multi-output）

  • 每个层级一个分类头（粗到细）。
  • 关键点不是“多输出”本身，而是：上层输出/特征如何影响下层（否则只是多任务并列，不一定强约束）。

主流套路 2：树结构/可扩展结构（tree-structured classifier）

  • 把分类器结构做成树（新增类只扩一条支路）。
  • 优点：扩展性强；缺点：和检测 head 的集成成本更高。

主流套路 3：图结构/embedding/metric learning（隐式编码层级）

  • 不一定改 head 成树，而是把层级关系融入特征空间或图传播。
  • 优点：表示能力强；缺点：实现复杂、和 YOLO 训练管线耦合更重。

hYOLO 的位置（你后面 Methods 会看到）： 它走的是 “多层级输出 + 逐级融合（coarse-to-fine fusion）” 这条路线，但把它严格落在 YOLOv8 的 head 上，并做了多个融合变体（V1~V6）来比较。

B. Loss 改法（loss-level）

怎么让“错得远”更痛？

你可以把层级 loss 分成两类工程实现方式：

  1. 距离/严重性加权（distance/severity weighting）
    • 需要定义层级距离：同父类距离小、跨大类距离大。
    • 然后把这个距离变成权重，乘在分类损失上。
  2. 层级一致性约束（hierarchy consistency regularization）
    • 强制“如果你在细类上很自信，那它的父类也应该合理/一致”。
    • 常见实现是加一个额外正则项，或用 mask/指示函数限制非法子类。
  3. 软标签 / 课程学习（更偏训练策略）
    • 软标签：给同父类兄弟类一点点概率；
    • 课程学习：先粗后细逐步训练。

hYOLO 的位置： 它更像第 2 类：通过一个由 α 控制的惩罚/正则，让模型在层级上更一致，并且把这个机制嵌进 YOLO 的端到端训练（而不是单独做分类网络）。

C. 指标与评估（metric/evaluation）

怎么“量化部分正确”？

层级指标你复现时要抓住两个核心点：

  1. 祖先集合型指标（ancestor-set based）
    • 对 leaf 预测类别 C_p 和真值 C_t，取它们的祖先集合 Ancest(C_p)、Ancest(C_t)，用交集/并集算层级 precision/recall/F。
    • 直观效果：同一子树内的错会得到更高分。
  2. 距离/成本型指标（distance/cost based）
    • 定义树上距离或分支距离，算“错到多远”。
    • 适合你想把误差解释成业务成本时。

hYOLO 的位置： 论文明确强调：必须用层级指标才能看见收益，否则你用 flat mAP/accuracy 很可能把“错得更近”的改进淹没掉。

你现在该带着什么问题进入 Methods？

为了让你下一节读得更“有方向”，我建议你记住 3 个复现问题（每个都对应一块代码）：

  1. 架构： YOLOv8 head 里具体新增了哪些分支？每个层级输出张量形状是什么？融合是 concat 还是加和？在哪里融合？
  2. loss： α 那项到底怎么计算？用到了哪些 mask/指示函数？它作用在 logits 还是概率上？

Ⅲ Method

3.1 Hierarchical Architectures

这一节的目标很明确：先把“层级分类”这件事定义清楚（什么叫层级、用哪种层级分类器），再说明“我到底怎么把层级信息塞进 YOLOv8 的 head（尤其是分类分支）里”。

本节核心结论

hYOLO 把 YOLOv8 的“分类分支”改造成多层级（level-wise）的分类头：每一层都有自己的分类器（LCL），并把上一层的输出特征在某个位置 concat 到下一层，让粗粒度预测作为细粒度预测的上下文。

关键概念/符号表

  1. Hierarchy（层级/类层次）：满足 root、父子关系、无环、反自反等条件的类别组织结构
  2. Tree vs DAG：树保证 root→任意节点路径唯一；文中选择 tree-based classifier（更简单直观）
  3. LCN：每个节点一个二分类器
  4. LCPN：每个父节点一个多分类器
  5. LCL（Local Classifier per Level）：每个层级一个多分类器（本文选用）
  6. hierarchy depth / levels（层级深度/层数）：有多少个 level 就复制多少份 backbone 输出给 head 用
  7. Insertion point（插入点）：把“层级分支/层级层”插到分类分支的哪个位置
  8. Concatenation point（拼接点）：把上一层的信息 concat 到下一层的哪个位置

逻辑分解

  1. 先定义“层级分类”问题：层级结构 + 走法（classifier 设计）
  • 文中把 taxonomy 叫做 hierarchy，需要满足 root、父子继承关系、无环、反自反等约束；并指出常见结构是 DAG 或 Tree，他们选 Tree（实现/理解成本更低）。
  • 接着讲 LCN/LCPN/LCL 三种“走法”，最终选择 LCL：每个层级一个多分类器，理由是降低复杂度、提升泛化。

  • 但它做了一个很关键的工程选择：当前层的预测，不被强制限制在上一层预测到的子集合里，这样模型在更深层还能“纠错”（避免上一层错了就一路错下去）。

跟你现在的 COCO 实验直接对应：

  • 你设置 hier_depth=2，就是 level0=12(super)、level1=80(leaf) 两个 LCL 分类器并行训练（每层各算各的分类 loss）。
  • 同时 level1 并不会被硬限制只能在 level0 预测的大类子集中选（至少在 3.1 这里，作者明确说“不限制”）。
  1. 再说明“改 YOLOv8 哪儿”：只改分类分支，不动回归分支
  • 作者强调：YOLOv8 head 负责输出 bbox、conf、class；bbox 回归流水线不变，只改分类路径来塞层级结构。 这对复现很重要：你不需要重写整个 Detect，只需要围绕 cls 分支做“多层 + concat”的结构扩展。
  1. 六种架构版本：核心变量只有两个

作者系统地试了 6 个版本，变化只来自两件事：

  1. Insertion point：层级分支插入分类 head 的哪个层
  2. Concatenation point：上一层输出在下一层的哪个位置 concat 其动机是：找一个既“信息量大”又“反向传播梯度传得好”的融合点。并说明哪些版本插入得更早/更晚。
  1. 重点讲 Version 4：它为什么长这样

作者在 3.1 里重点解释 Version 4（因为实验里它最好）。它的描述可以直接翻译成你复现时的“前向结构”：

(a) backbone 输出复制成 level 份

  • 举例三层 hierarchy：level0/1/2；那么 backbone 的输入特征会被复制三份，分别喂给每个 level 的分类分支。

(b) level 0：完全像原版 YOLOv8 一样做 flat 分类

  • level0 的 Conv2D 输出通道数 = level0 的类别数（作者举例 level0=2 类），这一步没有可用的上级信息，所以就是普通 flat 分类。

(c) level 1：从 backbone 来一份特征，再把 level0 的信息融合进来

  • level1 也从 backbone 取输入；但在 level1 的某个 Conv2D 之后，会把 level0 的特征信息 concat 进来，让 level1 的预测带上 level0 的上下文。

(d) level ≥1：加一个额外 Conv2D 做“融合后精炼”

  • Version 4 的一个关键点：从 level1 开始，每层都会多加一个 Conv2D，用来“消化”concat 后的混合特征（backbone 原始特征 + 上一层传来的层级信息），再去适配该层更细的类别数。

你可以把它理解成： concat 只是把信息拼在一起，但“拼一起 ≠ 融合好”。多出来那层 Conv2D 就是专门负责把上下文揉进当前层表征里。

PyTorch / Ultralytics 复现映射（你写代码时怎么落地）

你需要改的模块位置（高层定位）

  • 目标位置：YOLOv8 Detect head 的 cls 分支（论文说 bbox 分支不动）。
  • 结构形态：把原来 cls_head 的输出从 “1 个 logits（nc）” 变成 “L 个 logits（nc_level0, nc_level1, …）”，并在相邻 level 间做 concat + refinement conv。

最小可复现的前向伪结构（以你 COCO 两层为例）

  • backbone/neck 给出每个尺度的特征 F（YOLOv8 是多尺度，但先把它抽象成一份特征流即可）
  • F0 = F，F1 = F（复制两份，对应 hier_depth=2）

3.2 Hierarchical Models Evaluation Metrics

Hierarchical Models Evaluation Metrics 的核心是在讲：普通 precision/recall/F1 不知道“错得近还是错得远”，所以作者采用 Kiritchenko et al. (2006) 的层级指标，用“共同祖先有多少”来给“部分正确”记分。

本节核心结论

用“祖先集合交集”来定义层级版 Precision/Recall/Fβ，使得同一子树内的误分类罚得更轻、跨远子树罚得更重，从而能反映层级分类真正想优化的目标。

关键概念/符号

  • Ct：ground-truth class（真值类）
  • Cp：predicted class（预测类）
  • Ancest(C)：节点 C 的“祖先集合”（论文这里的定义很关键：是 root 到 C 路径上的所有节点，但不包含 root；并且从例子看它包含节点自身）
  • PrecHier / RecHier / Fβ,Hier：层级版 precision / recall / Fβ
  • β：Fβ 的权重参数，默认 β=1 时就是 F1Hier

指标定义

  1. 为什么要通过“祖先”来评估？(核心思想)

在传统的“扁平”分类（Flat Classification）中，预测错了就是错了。

  • 🍎 比如：把“苹果”预测成“梨”，或者把“苹果”预测成“汽车”，在传统指标里，这两种错误的惩罚是一样的（都是 0 分）。

但是在 层次化分类（Hierarchical Classification） 中，我们希望模型能通过层级关系获得“部分分数”：

  • ✅ 把“苹果”预测成“梨”：它们都是“水果”，所以错得不远，应该给一点分。
  • ❌ 把“苹果”预测成“汽车”：完全不相关，应该给更低的分（或 0 分）。

为了实现这一点，论文引入了 Kiritchenko 等人 (2006) 提出的基于 “祖先节点集合” (Set of Ancestors) 的评估方法。

🔑 关键概念：祖先集合 Ancest(C)

公式的基础是集合论。对于层级树中的任意一个类别节点 C ，它的祖先集合 Ancest(C) 包含了从该节点一直往上追溯到根节点路径上的所有节点（论文中规定不包括根节点 ）。

  • 直观理解：你可以把这个集合想象成该类别的“家谱”或“完整路径”。
  1. 细致拆解每一个公式 📐

公式 (1)：层次化精确率 (Hierarchical Precision)

PrecHier=|Ancest(Cp)∩Ancest(Ct)||Ancest(Cp)|
  • Cp : 预测的类别 (Predicted Class)。
  • Ct : 真实的类别 (True/Ground-truth Class)。
  • ∩ (交集): 代表预测路径和真实路径中重合的部分（即模型预测正确层级节点）。
  • |⋅| : 代表集合中元素的数量。

🧐 它是怎么来的？

传统的精确率是

预测正确的数量预测的总数量

在这里，我们将“数量”的概念替换成了“路径上的节点数”：

  • 分子：预测路径中，有多少个节点是和真实路径重合的？（做对的部分）
  • 分母：模型总共预测了多少个节点？（预测的总深度）

公式 (2)：层次化召回率 (Hierarchical Recall)

RecHier=|Ancest(Cp)∩Ancest(Ct)||Ancest(Ct)|

🧐 它是怎么来的？

传统的召回率是 预测正确的数量真实存在的总数量 。

同样地：

  • 分子：预测路径中，有多少个节点是和真实路径重合的？（做对的部分）
  • 分母：真实路径实际上有多少个节点？（应该预测出的总深度）

公式 (3)：层次化 F-Score

Fβ,Hier=(β2+1)⋅PrecHier⋅RecHierβ2⋅PrecHier+RecHier

🧐 它是怎么来的？

这是标准的 F-Measure 公式（精确率和召回率的调和平均数），直接套用了上面的层次化版本。当 β=1 时，就是我们最常用的 F1Hier ，代表精确率和召回率同等重要。

  1. 实战运用：图解计算 (Application) 🧮

为了真正掌握它，我们用论文 Figure 4 中的例子来算一遍。

假设：

  • 真实类别 ( Ct ) 是 L。
  • 根据图 4，L 的路径是 Root→B→F→L 。
  • 根据定义（排除根节点）， Ancest(L)={B,F,L} 。
  • 分母 |Ancest(Ct)|=3 。

我们将计算三种不同预测情况下的 F1 分数（假设 β=1 ）：

✅ 情况 1：预测在同一个子图 (Same Subgraph)

  • 预测 ( Cp )：M (L 的兄弟节点)。

  • 预测路径： Root→B→F→M 。

  • 预测祖先集合： Ancest(M)={B,F,M} 。分母为 3。

  • 交集 (重合部分)： {B,F,L}∩{B,F,M}={B,F} 。

    • 这意味着模型虽然在最后一层猜错了（猜成 M），但在前两层（B类和F类）是完全正确的！

  • 计算：

    • 交集数量 = 2。
    • Prec=2/3 , Rec=2/3 。
    • F1=2/3≈0.66 。

结论：传统分类器会给 0 分，但这里给了 0.66 分，奖励了它“虽错但近”。

⚠️ 情况 2：预测在稍远的子图 (One Common Ancestor)

  • 预测 ( Cp )：N。

  • 预测路径： Root→B→G→N 。

  • 预测祖先集合： Ancest(N)={B,G,N} 。分母为 3。

  • 交集： {B,F,L}∩{B,G,N}={B} 。

    • 只有第一层 B 是对的，后面全错了。

  • 计算：

    • 交集数量 = 1。
    • Prec=1/3 , Rec=1/3 。
    • F1=1/3≈0.33 。

结论：分数降低了，反映出这个错误比上一个更严重。6

❌ 情况 3：完全不相关 (No Common Ancestors)

  • 预测 ( Cp )：P。

  • 预测路径： Root→C→H→P 。

  • 交集： {B,F,L}∩{C,H,P}=∅ (空集)。

  • 计算：

    • 交集数量 = 0。
    • F1=0 。

结论：这就是“把苹果当成汽车”，零容忍。

把它映射到你正在做的 COCO 12→80 两层实验（很重要）

你现在的层级深度是 2：

  • level0：12 个 super 类
  • level1：80 个 leaf 类

在这种 两层树 里（假设 root=“all”不计入祖先集合），通常：

  • 对一个 leaf 类：Ancest(leaf) 会包含 {super, leaf}（长度 2）
  • 对一个 super 类：Ancest(super) 会包含 {super}（长度 1） （这和论文例子“祖先集合包含节点自身、但不含 root”的写法一致）

于是你会看到一个很直观的效果：

  • 如果 leaf 预测错了，但还在同一个 super 下面： 交集至少还有 {super}，所以分数不会掉到 0（“错得近”）。
  • 如果 leaf 预测错到别的 super： 两边可能只剩下 very high-level（甚至没有）共同祖先，分数会更低甚至 0（“错得远”）。

这正好解释了为什么你做 hYOLO 实验时，除了看普通 F1/mAP，还要看 F1Hier：它能告诉你“模型是不是把错误往正确大类里收敛”，而这正是层级训练+层级惩罚项想实现的目标。

复现落地：你评估代码该怎么写（最小实现逻辑）

按论文定义，你的评估流程（对每个 matched 的预测框/分类结果）基本就是：

  1. 从 hierarchy（你现在的 yaml/映射表）拿到：Ancest(Cp) 和 Ancest(Ct)
  2. 求交集大小 k = |Ancest(Cp) ∩ Ancest(Ct)|
  3. 算 PrecHier = k / |Ancest(Cp)|，RecHier = k / |Ancest(Ct)|
  4. β=1 ⇒ F1Hier = 2*P*R/(P+R)

你下一步如果要把它和 Ultralytics 的评估接上：核心就是“先完成检测匹配（TP/FP），然后在 TP/FP 的分类结果上算层级指标”。论文这一节讨论的是分类指标本身，匹配策略属于你接到 YOLO 的工程实现部分。

3.3 Modified Loss Function

下面进入 3.3 Modified Loss Function。这一节你只要抓住两件事：(1) 他们把 YOLOv8 的 loss“复制到每个层级”再做平均；(2) 从第 2 层开始，在分类 loss 里加了一个“层级违规惩罚项”，用 α 控制强度。

本节核心结论

hYOLO 的总 loss = 各层级（level-wise）检测 loss 的平均值；并且在 level>0 的分类项里额外加入“非父子一致”的惩罚，让模型更倾向于把概率质量放到“父类允许的子类集合”里。

YOLOv8 原始 loss

YOLOv8 的检测 loss 可以概括为：

  • 总 loss

    L=LReg+LCls

  • 回归项（bbox）

    LReg=wboxLCIoU+wdflLDFL

  • 分类项

    LCls=wclsLBCE

这里的 wbox,wdfl,wcls 是 YOLOv8 默认的三个权重（论文里给了默认值），你可以把它理解成：“框回归重要程度、边界分布回归重要程度、分类重要程度” 的配比。

第一处改动

每个层级都算一份 loss，然后平均

他们引入层级后，不再只有一个分类输出，而是 每个层级都有自己的分类输出

于是训练时：

  1. 对每个层级 i 都计算一份 “检测 loss”： L(i)=LRegression(i)+LClass(i) 其中回归仍是 CIoU+DFL，分类仍是 BCE（结构不变，只是“多了一套输出/多了一套计算”）。
  2. 总 loss 取平均（论文写得很明确，就是 average）： Ltotal=1l∑i=0l−1(LRegression(i)+LClass(i)) 这里 (l) 就是 hierarchy depth（层级数）。

结合你现在的实验：hier_depth=2 等价于：level0 的 loss + level1 的 loss，除以 2。 所以你会看到一个很直观的现象：层级越深，总 loss 的数值尺度会被平均“摊薄”，这也是为什么他们强调每层用同一套权重，保证每层对总 loss 的贡献可控。

第二处改动

在 level>0 的分类 loss 里加“层级违规惩罚”

这部分就是你后面做实验解释 α 的关键。

  1. level0：不加惩罚（因为没有父类）
LCls0=wclsLBCE0
  1. level l（l>0）：BCE + 惩罚项（α 控制强度）

论文给的形式是：

LClsl=wcls(LBCEl+α∑i=1S(1−δil)⋅confil)

各符号怎么理解（复现时非常重要）：

  • S ：该层级的类别数（比如你 level1 是 80）
  • confil ：当前层 l 对类别 i 的 置信度（直觉上就是 sigmoid 后的概率/score）
  • δil ：
    • 若 “类别 i ” 是 上一层预测父类 的子类 → δil=1
    • 否则（不属于那个父类的子类集合）→ δil=0

所以 (1−δil)⋅confil 的含义就是：

只要你在当前层给“非法子类”（不属于上一层父类的子类集合）分了概率质量，就要被罚；而且你分得越自信（conf 越大），罚得越重。

并且论文明确说：

  • α=0 → 惩罚项消失 → 退化回普通 BCE（也就是不做层级一致性约束）
  • α 越大 → 越“强迫”模型遵守父子一致性

用你 COCO(12→80) 两层来直观理解这个惩罚

  • level0 预测的是 12 个 super 类（比如 “vehicle”）
  • level1 预测的是 80 个 leaf 类（比如 “car/bus/truck …”）

如果 level0 预测 “vehicle”，那么 level1 合法的子类集合应该是车、公交、卡车等。

  • 如果 level1 给了 “cat/dog” 很高置信度 → 这些类对当前 parent 来说是 非法子类 → 它们对应的 confil 会被加进惩罚项里 → 梯度会推动模型 把这些非法类的概率压下去（尤其是高置信度的那种）

这件事和 3.2 的层级指标是对齐的： 3.2 用指标奖励“错得近”；3.3 用 loss 在训练阶段就把“错得远”变得更痛。

PyTorch/Ultralytics 复现映射

你应该怎么实现这项惩罚（最小可行版本）

你可以按下面这个“最小实现思路”落地（先别管多尺度，先把单个 feature map/单批次想清楚）：

  1. 预处理层级关系
  • 为每个 level>0 建一个映射：children_of_parent[parent_id] -> [child_ids...]
  • 或者直接做成一个矩阵 M：形状 [num_parent, num_child]，合法 child 位置为 1
  1. 前向得到两层的分类 logits
  • logits_parent: shape [N, num_parent]
  • logits_child: shape [N, num_child]
    • 这里 N 可以理解成 “所有参与分类计算的预测点/正样本数”（YOLO 里通常是匹配到 GT 的那些点）
  1. 计算 parent 预测
  • parent_id = argmax(sigmoid(logits_parent), dim=-1) → shape [N]
  1. 构造当前样本的合法 mask
  • valid = M[parent_id] → shape [N, num_child]（每个样本一行合法子类 mask）
  1. 算惩罚项
  • conf_child = sigmoid(logits_child)
  • penalty = ((1 - valid) * conf_child).sum(dim=-1).mean()
  • L_cls_child = w_cls * (BCE_child + alpha * penalty)
  1. 总 loss：按层平均
  • L_total = (L_level0 + L_level1 + ...)/hier_depth

注意：论文的写法看起来是“对所有类别求和”，所以惩罚更像“把非法概率质量整体压下去”，而不仅仅罚 argmax 那一个类。

Ⅳ Experiments

4.1 Experimental Variants of Hierarchical Architectures

论文 4.1 Experimental Variants of Hierarchical Architectures 这一节做的事情很“工程化”：固定“层级检测”的大框架不变，只系统比较 6 种“信息从上一级怎么接到下一级”的接法（V1~V6），看哪种融合点最合理。

下面我按你说的“分段节奏”来讲（但会把关键结论在这一轮就讲完整，方便你直接对照代码/参数）。

段 1

4.1 先把“对比实验在比什么”讲清楚（你读这一段要抓的主线）

他们在比两件事：Insertion point & Concatenation point

作者说 6 个版本都只在两处做差异：

  1. hierarchical branch 插入点（insertion point）：从分类分支的哪一层开始“分出”一个层级分支；
  2. 拼接点（concatenation point）：把“上一层级的输出”在当前层级的哪一层进行 concat。

而所有版本都遵循同一个骨架流程：

  • YOLOv8 backbone 的输出 先复制成 = 层级数（hierarchy levels）份；
  • 每个 level 都有自己的分类分支；
  • 并且在某个指定位置，把 level-1 的信息 concat 到 level。

图 5/图 6 里的红色虚线箭头就是“跨层级连接”：箭头起点=插入点，箭头终点=拼接点。

为什么“融合太早/太晚”都会坏？

作者给了一个非常典型的深度学习直觉（你后面做 ablation 解释时直接用）：

  • 太晚融合：低层（边缘/空间）信息可能已经丢掉，空间上下文不足；
  • 太早融合：低层噪声太多，会“压垮”分类器，让高层语义学不起来。

这就是 4.1 的核心：找一个“刚刚好”的融合时机。

段 2

V1 & V2（作者先讲“早融合”的坑，再讲“稍晚一点”的改进但不稳）

Version 1：branch 插得还行，但 concat 太早

作者对 V1 的判断非常明确：

  • 插入点：level0 在 Conv2d 之后引入层级结构（此时已经过了一些 conv，开始有“有意义的表示”）；
  • 问题：但是把 level0 的信息 concat 到 level1 的位置 太早——直接在 backbone 输出后就拼，导致“过早融合”打乱了更高阶特征的形成，于是平均 F1Hier 更低。

你复现实验时，这句话等价于：“把 coarse 的信号塞到 fine 分支里没错，但塞的时间点不对，会害 fine 分支学不到该学的东西。”

Version 2：保留 insertion，但把 concat 往后挪

V2 的改动很“精准”：

  • insertion 仍然和 V1 一样：level0 的 Conv2d 之后；
  • 但 concat 往后挪：在 level1 的 Conv2d 之前再拼，这样 level1 先自己学一段，再接上 level0 信息。

结果：整体 F1Hier 略涨到 0.965，但 worst-class F1Hier 变低（0.738），作者解释为：延迟融合可能让某些困难类别拿不到关键互补的低层特征（或者说，关键上下文来得太晚）。

段 3

V3~V6（开始围绕“更合理的融合时机 + 融合后怎么消化”做文章）

评估怎么选“最优结构”？

作者先说明他们怎么评：

  • 用 F1Hier 在每个 level 上对比（level 越深越细，越难，所以从 level0→level3 都会下降）；

  • 选最优结构看两条：

    1. level3 的 F1Hier（包含完整类别集合、最细粒度、最关键）；
    2. level3 的 worst-class F1Hier（看“最难类”有没有被照顾到）。

这两条非常重要：它直接解释了为什么论文最终选 V4——它不是只看平均，还看“最差类”。

Version 3：把 insertion 和 concat 都往更早的阶段移动

动机：让每个层级更早开始处理“带上下文的特征”，在更靠近融合点的地方一起学“空间 + 语义”的协同表示。

结果：Table 1 显示 V3 相比前面版本，平均和 worst-class 都更好（作者明确说是 improvement）。

Version 5：沿用 V3 思路，但整体“再往深一层”——反而退步

作者说 Version 5 保留 V3 的插入与拼接方案，但两个点都 往网络更深处移动一个卷积层。结果整体和 worst-class 都下降，推测是融合更晚导致 fine-grained 表征不够“可分”。

Version 4 & Version 6：关键新招——“融合后加一层 Conv2D 做 refinement”

作者点出 V4 和 V6 的核心结构差异：

  • 从 level1 开始每层多加一个 额外 Conv2D，作为 refinement block：不是简单把上一级信号“传过去”，而是做一次再加工，用来解决融合后可能出现的冲突/错位表示，并过滤噪声冗余。

这点你可以直接映射到你代码理解：concat ≠ 融合完成，还需要一个“消化层”。

Version 6：再加“多处 concat（skip-like）”但效果反而不好

V6 进一步引入多个 concat 点：例如从 level0 直接连到 level1 和 level2，想让高层获得更丰富的低层信息（类似 skip connections 的直觉）。但结果并没提升，整体不变、worst-class 反而更差。

Table 1 的关键数字：为什么最终选 V4？

Table 1 给的是 100-class 的 V1 数据集上，各版本在各层级的 PrecHier/RecHier/Avg F1Hier，以及 level3 的 worst-class F1Hier。你读表只需要盯住 level3 两列即可。

我把最关键的 level3 摘出来（论文表里的数字）：

  • V1：level3 F1Hier = 0.9630，worst = 0.779
  • V2：level3 F1Hier = 0.9650，worst = 0.738
  • V3：level3 F1Hier = 0.9700，worst = 0.800
  • V4：level3 F1Hier = 0.9710，worst = 0.814 ← 最强、且最稳
  • V5：level3 F1Hier = 0.9680，worst = 0.744
  • V6：level3 F1Hier = 0.9680，worst = 0.701

最后作者补了一句“综合判断”：各版本 precision/recall 差异不大、level0 往往都很高，但 V6 在 level0 recall 有下降；整体上 V4 在各层级更均衡、并且 overall + worst-class F1Hier 都最好，所以选 V4 作为最优结构。

复现

把 4.1 和你当前的复现实验对齐（你现在该怎么用这节）

你现在在跑的是 COCO 12(super) → 80(leaf) 的两层层级实验。4.1 给你的直接启发是：

  1. 你要做“结构 ablation”时，优先对照 V3/V4/V5 这条线：它们是在“融合时机”附近做小步移动，最适合解释“插入/拼接早一点或晚一点到底影响什么”。
  2. 如果你发现“concat 了但效果没提升”，很可能就是论文强调的：需要 V4 那种 refinement conv 来消化融合后的表示。
  3. 评价不要只看平均，还要看“最难类/最差类”或类似的鲁棒性指标（你可以在 COCO 上用“哪些类最差”来复刻这种思路）。

4.2 Datasets

According to a document from 27 Oct 2025, 论文 4.2 Datasets 这一节其实是在把他们的实验数据“怎么来的、长什么样、为什么这样设计”交代清楚——因为 hYOLO 的核心卖点之一就是：在超大类数（1011 类）、而且类之间视觉相近的“超市商品”场景里，层级信息更有价值。

数据集到底是什么

  1. 规模与划分
  • 总计 78,852 张图像
  • 1011 个 grocery store items（= 1011 个类别）
  • 划分为：
    • 训练集 42,456
    • 验证集 18,198
    • 测试集 18,198

这告诉你：他们不是 COCO 那种“自然场景随手拍”，而是一个类别数极多、但每一类都很像商品 SKU 的识别任务。

  1. 图像是怎么“合成/增强”出来的（关键）

作者描述了一套很具体的“图像增广/构图流程”（你可以把它理解成：用多物体合成的方式制造检测数据）：

  • 对于某个商品的一张曝光图（exposition），随机选择 17 个其它商品的曝光图
  • 把这些商品按从大到小的顺序，一个个叠加到同一张图里
  • 每次叠加都要检查遮挡：最大遮挡不超过 70% 才允许加入

这一步的含义很重要：

  • 每张图不是只有一个商品，而是“一个主商品 + 一堆干扰商品”，天然就更贴近货架/购物篮那种复杂场景。
  • “从大到小叠加 + 遮挡上限”是在控制难度：既有遮挡和拥挤，但不至于完全看不见。
  1. 一张图平均有多少个物体？数据是否均衡？

论文给了两个你很该记住的统计：

  • 平均每张图约 18 个物体（因此是“多目标检测 + 细粒度分类”压力很大）。
  • 数据集是均衡的：每个类在
    • 训练集约 700 个实例
    • 验证和测试各约 300 个实例

并且这些实例来自对每个商品拍摄的 **10 种不同曝光（expositions）**做排列组合生成。

你可以这么理解：

  • “均衡”让实验更像在验证“结构/损失/层级本身是否有效”，而不是被长尾分布掩盖。
  • 10 种曝光 + 组合生成，既增加外观变化，也让每一类有足够多训练样本。
  1. 为什么还要搞一个 “100 类子集”？

作者很直白：为了省时间和算力，他们用一个 只包含 100 类的 reduced dataset 来跑部分实验（尤其像你刚读完的 4.1 那种要比较 V1~V6 的结构消融）。这个 100 类子集被采样为对完整数据集“统计上具有代表性”。

这解释了你读 4.1/表 1 时的疑问：为什么不直接在 1011 类上跑 6 个结构？ 因为那样代价太高，所以先用“代表性的 100 类”快速筛结构。

把 4.2 和你当前 COCO(12→80) 实验对齐：你应该得到什么启发

你现在是在 COCO 上做 2 层层级（12 super → 80 leaf）。而论文的数据集有几个“更偏向 hYOLO 优势”的特征：

  • 类数超级多（1011）：flat 分类更容易“错得离谱”；层级就更有用。
  • 类别视觉相近（都是商品）：层级里的“视觉相似性”会更强烈地影响效果（他们后面 4.3 也会专门讲 hierarchy v1/v2）。
  • 多物体、遮挡多：检测与分类耦合更紧，层级上下文可能更能稳定分类分支。

所以你在 COCO 上做实验时，一个很合理的预期是：

mAP/F1 可能不一定大涨，但层级指标（比如 F1Hier、以及“错在同一 super 的比例”）更容易体现收益。

4.3 Dataset Hierarchical Structures

4.3 Dataset Hierarchical Structures 这一节的主线很清楚：作者在同一套 1011 类商品数据上手工设计了两种“类层级树”，用来专门检验两件事——(1) 层级深度有多重要；(2) 层级是否应该体现“视觉相似性”。

核心

为什么要做两种 hierarchy？

他们不是随便分组，而是把 hierarchy 当成一个“可控变量”来做消融：

  • 如果 hierarchy 只是“商店货架分类”（语义/管理方便），但不反映视觉像不像，那它对模型的 coarse-to-fine 引导可能有限；
  • 如果 hierarchy 把视觉一致的属性放在更靠上的层级（更早决策），就可能更能帮助细粒度分类。 这就是 Version 1 vs Version 2 的动机。

Dataset Hierarchy Version 1

按“商店品类”建树（不管视觉像不像）

Version 1 的定义一句话概括就是：

类关系完全基于商店类别（store categories），刻意忽略物体之间的视觉相似性。 作者说这么做主要是“方便”，让层级看起来像真实超市的分类：从大类到小类逐步细化。

你可以把它理解成：语义/业务 taxonomy，比如“食品→零食→饼干→某品牌某口味某规格”。

Dataset Hierarchy Version 2

把“视觉/语义相似性”显式塞进层级里（5 层）

Version 2 的思路是反过来：先把更稳定、视觉更一致的属性放在更低层（更靠近输入、也更“粗”），再逐步走向商品语义与品牌。

论文给了一个非常具体的“从上到下（或从粗到细）”设计链：

  • Level 0：形状 shape（例如 bag, bottle, box, can, cylinder 等）

  • Level 1：大小 size（例如 1000g、1000ml、100g、10g 等）

  • 更高层：回到商店命名体系

    • 例如某一层用 type（toothbrush, water, flour 等）
    • 最后一层用 brand / manufacturer

  • 并且作者强调：最高层的类别命名与商店库存系统完全一致（最终叶子类就是实际 SKU/库存命名）。

同时，Version 2 是一个5 层的层级结构（论文图 8 标注 “five hierarchical levels”）。

直觉上： 先问“它长得像瓶子还是盒子？” → 再问“它是多大规格？” → 再问“它是什么类型的东西？” → 最后才问“它是哪家品牌的哪一个具体产品？” 这比 Version 1 那种“先按货架语义分大类”更贴近模型的视觉学习路径。

两个版本的对比

两个版本的“深度”也不同：V1 更浅，V2 更深

从后面实验表（Table 2）能看出来：

  • Dataset Version 1 评到 level 3（也就是 0~3 共 4 层）
  • Dataset Version 2 评到 level 4（也就是 0~4 共 5 层）

这正好对应 4.3 一开始说的：他们要同时比较层级深度与视觉相似性编码的作用。

作者想传达的“结论味道”（为后面 4.5 铺垫）

虽然正式对比在 4.5，但论文已经提前点明 Version 2 的价值：它考虑了视觉与语义相似性，在 100-class 子集上表现更好（precision/recall/F1、TP conf 更高、FP conf 更低）。

复现

对你复现/做 COCO 12→80 两层实验的启发（非常实用）

你现在的 12→80 属于“语义大类→细类”。4.3 给你的关键启发是：

  1. Hierarchy 不是越“符合人类语义”越好 Version 1 就是“符合超市分类”，但作者明确说它不管视觉相似性。

  2. 更有用的 hierarchy 往往是：上层节点对应“视觉更稳定的共同属性” 他们甚至把 shape 放在最底层（level 0）当作最可靠的区分特征，再往下才是 size/type/brand。

  3. 你的 COCO 两层可以做一个很干净的对照实验：

    • Hierarchy-A：你现在的 12 super（纯语义）
    • Hierarchy-B：尝试用“视觉相似性/外观聚类/混淆矩阵聚类”生成 12 个上层组 然后用同样的训练设置对比：看层级指标、以及“错分是否更常落在同一上层组”（论文后面也会用类似思想分析）。

4.4 YOLO Model and Hyperparameters

According to a document from Oct 27, 2025（hYOLO 论文 arXiv 版本），4.4 YOLO Model and Hyperparameters 这一节主要是在回答：他们把 hYOLO（V4 架构 + 3.3 的改造 loss）接到 YOLOv8 之后，训练时哪些设置保持默认、哪些必须改，以及为什么要改。

架构选择

  1. 用的是什么 YOLO？怎么保证不同结构对比公平？
  • 他们把 3.1 的 V4 架构和 **3.3 的改造 loss（Equation 7）**都实现进了 YOLOv8。
  • 为了更快收敛、泛化更好、并且保证不同架构/设置可公平比较，他们统一采用 COCO 预训练权重作为所有模型的相同起点（standardized starting point）。

对你来说，这一条直接解释了：为什么你训练时从 yolov8n.pt / yolov8*.pt 开始是合理的——这就是作者的实验范式。

超参数

  1. 哪些超参不动？哪些超参动了？

作者明确说：大部分超参保持 YOLOv8 默认，但他们刻意改了一个非常关键的权重：

只改了：分类损失权重 (w_{cls})

  • YOLO 的总损失由 **CIoU（box）+ DFL + BCE（cls）**组成；默认权重分别是

    • wbox=7.5 （CIoU）
    • wdfl=1.5 （DFL）
    • wcls=0.5 （BCE 分类）

  • 作者强调：这些权重应该任务相关。比如自动驾驶更偏检测定位，可以提高 wbox ；但他们的应用（超市商品细粒度识别）更看重“分对类别”，所以要提高 wcls 。

他们怎么选的 wcls ？（Figure 9 的结论）

  • 他们在 100-class reduced dataset 上实验增大 wcls ，观察 test F1：

    • 从 ( wcls =0.5) 时的 0.948
    • 提升到 ( wcls =2) 时达到峰值 0.962

  • 所以把默认 0.5 改成 2。

这条和你现在配置里看到的 cls: 2.0 是一致的（在他们附录给的 default.yaml 改动列表里也能直接看到 cls: 2.0）。

优化器

  1. 优化器怎么选？

作者还补充：他们更偏向 SGD 而不是 Adam，理由是对他们的模型与数据集来说训练更稳定（stability 更好）。

复现

  1. 把 4.4 直接对齐到你现在的 COCO(12→80) 实验：你该怎么理解“自己在调什么”

你现在的任务（COCO 80 细类 + 12 super 类）其实也偏“分类正确性很重要”，所以 4.4 给你一个很明确的实验抓手：

  • 想让层级更“发挥作用”，除了调 α（3.3），还要关注 **分类损失权重 ** wcls
    • wcls 小：模型更愿意把容量用在定位/回归上
    • wcls 大：模型更愿意把容量用在“分对类别（尤其细粒度）”上
  • 作者在商品识别里把 wcls 提到 2，就是为了匹配“分错类比框偏一点更糟”的目标函数。

4.5 Model Training and Evaluation

According to a document from 27 Oct 2025（hYOLO 论文 arXiv v1），4.5 Model Training and Evaluation 做了三件很关键的事：定最终架构（V4）→ 统一训练设置跑两种层级（V1/V2）→ 用“F1 + TP/FP 置信度”来解释模型到底“更准了”还是“更靠谱了”。

训练架构和数据集版本

  1. 他们怎么训练（Training）：架构固定用 V4，然后只换“数据层级版本”
  • V4 是从 4.1 的 6 个结构里选出来的，因为它在“全类平均 + 最差类”上都最好，所以在 4.5 里他们把 V4 当作最终架构固定下来。
  • 之后用同一个 V4，分别在 **4.3 的两种层级数据集版本（Dataset hierarchy V1 / V2）**上训练，然后比较结果。

你可以把这里理解成：模型结构先消融定型（V4），再把“层级怎么建”当作主要变量来实验。

评估指标

  1. 他们怎么评估（Evaluation）：不只看 F1，还看 TP/FP 的“置信度校准”

在 4.5 里，他们明确用 3 个指标评估：

  1. F1 score
  2. TP confidence（真阳性预测的平均置信度）
  3. FP confidence（假阳性预测的平均置信度）
  1. TP Conf (True Positive Confidence Score) ✅
  • 定义：它是针对某个类别，计算所有预测正确 (True Positive) 的结果的置信度分数的平均值 。
  • 直观理解：当模型答对的时候，它有多大的把握？
  • 期望：这个分数越高越好。这意味着模型不仅答对了，而且非常确信自己是对的。
  1. FP Conf (False Positive Confidence Score) ❌
  • 定义：它是针对某个类别，计算所有预测错误 (False Positive) 的结果的置信度分数的平均值 。
  • 直观理解：当模型答错的时候，它有多大的把握？（是不是在“一本正经地胡说八道”？）
  • 期望：这个分数越低越好。这意味着即使模型答错了，它也比较犹豫，置信度不高。这在实际应用中非常重要，因为我们可以通过设置一个置信度阈值（比如 0.5），把这些低置信度的错误过滤掉。

这一点非常“落地”：

  • TP conf 高代表“答对时更有把握”（更可信）
  • FP conf 低代表“答错时不那么自信”（更安全、可控）

实验结论

  1. 100-class 实验结论：层级版本 V2（考虑视觉/语义相似性）整体最好

作者结论写得很直接：Dataset hierarchy V2（把视觉与语义相似性编码进层级）在 100-class reduced dataset 上表现优于 V1，也优于 flat YOLOv8（在 precision/recall/F1 三项都更好）。

并且 Table 2 给出了关键现象：V2 往往能做到 更高 TP conf、且更低 FP conf（“更有把握地对、但错得更不自信”）。

层级设置

  1. 为什么 flat YOLO 要训练 5 个？——为了公平对齐每个层级

因为 flat YOLOv8 天然没有“层级输出”，作者为了对齐比较做了一个很重要的实验设计：

  • 对应 Dataset V2 的每个层级，单独训练一个 flat 模型（一共 5 个）
  • 同时由于“hierarchical F1”不适用于 flat，最终比较时统一用 标准 precision/recall/F1，保证评估口径一致。

这段对你做 COCO(12→80) 很有启发： 如果你想“严格公平”地证明层级训练有效，flat baseline 最好也要做这种“对齐层级标签空间”的对照，而不是只拿一个 80 类 flat 去比所有层级。

扩展到 1011-class

  1. 扩展到 1011-class：V2 的“语义一致性”更强，但 flat 的 F1 可能更高（代价是更危险的 FP）

当他们把实验扩展到 更复杂的 1011-class数据集时，作者观察到：

  • V2 依然体现出 更低 FP conf、更高 TP conf（更好的置信度校准）。
  • 他们还引入了一个很“层级味”的分析：“落在同一子图（same subgraph）的 FP 比例”，用于衡量“即使错了，错得是否还语义接近”。结果 V2 在最后层级 level 4 是 0.6844，flat 是 0.6508。

但作者也很诚实：flat YOLOv8 在某些情况下 F1 更高，只是伴随：

  • 更高的 FP confidence
  • 更可能产生语义无关的错分（same-subgraph FP fraction 更低）

这就是 4.5 最想传达的“应用视角”结论： 层级模型不一定让“纯 F1”永远最大，但能让模型更“靠谱”——错了也更不自信、更接近正确语义。

dependency loss 权重实验

  1. 4.5 后半段：开始扫 α（dependency loss 的权重），解释为什么 α=50 也合理

紧接着他们做了你接下来做实验最需要的 α 扫描（Table 4）：

  • 他们在 Eq.7 的惩罚项上测试多个 α，并解释：α=50 看起来很大，但其实合理，因为惩罚项有时会比 BCE 小一个数量级，不放大就几乎不起作用。
  • 不同 α 会“优化不同指标”：例如 TP conf 最好在 α=0.9；而 平均 F1 和 FP conf 最好反而在 α=0（不加惩罚）。
  • 更细的 trade-off：α 增大会让 F1 上升，但 TP conf 下降；不过它能显著改善“最差类别”的 F1（例如 worst-class F1 从 α=0.5 时的 0.362 提升到 α=25 时的 0.589）。

作者还提醒：只看 Avg F1 会误判——α=0 虽然 Avg F1 很高，但 worst-class F1 会明显更差（论文举例 worst-class F1=0.485），说明类别间表现不均衡；而中等 α（如 25）更均衡。

把 4.5 翻译成你接下来该干什么（对应你现在的 COCO 两层实验）

你现在已经具备复现实验的两个关键“旋钮”了：

  1. 层级结构怎么建（像 4.3 的 V1 vs V2）
  2. dependency loss 的权重 α（像 4.5 的 Table 4 扫描）

而 4.5 给你的“实验解释框架”是：

  • 别只盯一个指标：至少同时看 F1 + TP conf + FP conf（你就能解释“更准 vs 更靠谱”）。
  • α 不是越大越好：它会带来均衡性收益（worst-class F1 上升），但可能牺牲 TP conf；你需要用实验去解释这个 trade-off。

如果你愿意，我下一步就按论文的顺序，带你把 Table 2 / Table 3 / Table 4 变成你自己的“复现对照表”：在你的训练日志里应该对应看哪些字段、你该怎么记录每个 level 的 TP/FP conf，最后怎么写成一段像论文那样的结论。

表格解读

作者在实验部分用 Table 2 / Table 3 / Table 4 把 “层级怎么建、模型到底好在哪里、α 怎么影响 trade-off” 讲得非常集中。

在正式读表之前，先把作者的“评估口径”钉死：他们在 4.5 里说评估三件事——F1、TP confidence、FP confidence，并且 TP/FP conf 是“先按类求平均，再对所有类做平均”。

Table 2

100-class 数据集上，V1/V2 层级 vs Flat YOLOv8（都用 V4 架构）

这张表在回答什么？

在一个较小但代表性的 100-class reduced dataset 上，层级结构怎么建（V1 vs V2），以及 hYOLO vs flat YOLOv8 到底差在哪。作者还强调为了公平：flat 模型没有层级输出，所以他们训练了 5 个 flat 模型分别对应 V2 的各个层级，并统一用标准 precision/recall/F1 来比（不用 F1Hier）。

表头怎么读？

  • Model Type：Dataset Version 1 / Dataset Version 2 / Flat YOLOv8
  • Hier. Level：层级深度（越往下越细粒度，越难）
  • Precision / Recall / F1 Score：常规分类指标
  • TP Conf / FP Conf：正确/错误预测时的平均置信度（作者用它解释“更准 vs 更靠谱”）

把关键数字“读出来”

(A) Dataset Version 1（层级 V1，只有 level 0~3）

  • level0 F1=0.9961 → level3 F1=0.9558（越细越难，F1 下降明显）

(B) Dataset Version 2（层级 V2，level 0~4）

  • level3 F1=0.9763，level4 F1=0.9654；FP conf 在 level2/3 大约 0.0679/0.0683（整体偏低）

(C) Flat YOLOv8（用 V2 标签空间逐层训练/评估）

  • level2 F1=0.9801（这一层 flat 很强）
  • level4 F1=0.9635
  • FP conf 整体更高：例如 level2 FP conf=0.1002，level1 FP conf=0.1259（表里 level3 FP conf 甚至到 0.4123，明显比其它行高很多——至少能确定作者想表达“flat 的 FP 更自信/更危险”）

作者从这张表得出的结论是什么？

作者在正文里直接总结：V2（考虑视觉+语义相似性）在 100-class 上优于 V1 和 flat，尤其体现在：

  • TP conf 更高（答对时更有把握）
  • FP conf 更低（答错时不那么自信，更可靠）

你复现时读 Table 2 的“抓手”

  • 同一层级对比：比如 level2，V2 的 FP conf(0.0679) 明显低于 flat(0.1002)，TP conf 也更高一些（0.9457 vs 0.9374）。

Table 3

把难度拉满到 1011-class，V2(hYOLO) vs Flat，并加入“语义接近的错分”指标

这张表在回答什么？

在完整 1011-class 数据集上，层级模型的优势是否还在？如果 flat 的 F1 更高，那层级模型到底“赢在哪里”？

表头除了 Table 2 的那些列，多了一个：

  • FP Same Subgraph：错误预测（FP）里，有多少比例落在“与 GT 同一子图/同一大分支”里——用来衡量“错也错得更接近”。

关键数字怎么读？

(A) hYOLO + Dataset V2

  • level4：F1=0.8883，TP conf=0.8735，FP conf=0.0730，FP Same Subgraph=0.6844

(B) Flat YOLOv8

  • level4：F1=0.8931（略高于 hYOLO），TP conf=0.8710，FP conf=0.0970，FP Same Subgraph=0.6508

作者的解释（非常重要的“论文叙事”）

作者明确写了两点 trade-off：

  1. 层级模型（V2）：FP conf 更低、TP conf 更高，并且“错分更语义接近”（0.6844 > 0.6508）
  2. flat 模型：F1 更高，但代价是 FP conf 更高、错分更可能“语义不相关”。

这就是 hYOLO 想卖的核心价值：不只追 F1，还追“可用性/安全性/可解释性”——答错时别那么自信，错也尽量错在“同大类”。

Table 4

扫 α（dependency penalty 权重），把“平均效果 vs 最差类”讲透

这张表在回答什么？

3.3 的改造 loss（Equation 7 的 penalty）到底要给多大权重 α？α 改了会带来什么 trade-off？

作者先解释一个很多人会误解的点：α=50 看起来很大但合理，因为 penalty 项有时会比 BCE 小一个数量级；α 不放大，penalty 对总 loss 的影响几乎可以忽略。

表头怎么读？

每个 α（0 / 0.5 / 0.9 / 25 / 50），都分别报告 level0~4 的：

  • TP conf
  • FP conf
  • Avg. F1 Score
  • F1 Worst Class（最差类别的 F1：作者用它衡量“长尾/难类是否被照顾到”）

直接看最关键的 level4（最细粒度、最难）

  • α=0（不加 penalty）：Avg F1=0.9610 很高，但 Worst-class F1=0.4850（最差类很惨）
  • α=0.5：Worst-class F1=0.3620（更惨，说明“加一点点 penalty 但不够强/不够稳”可能反而扰动训练）
  • α=0.9：TP conf 表现最好（作者在正文点名 “best TP confidence at α=0.9”）
  • α=25：Avg F1=0.9557（略低于 α=0），但 Worst-class F1=0.5890（最差类大幅改善）
  • α=50：Worst-class F1=0.4950（回落一些），说明 penalty 太强也可能带来副作用

作者把这张表的“正确读法”写得很直白：

  • 不同 α 会优化不同指标：例如 α=0.9 最利于 TP conf，而 α=0 最利于平均 F1 和 FP conf（“best FP confidence when no penalty term was added”）
  • 只看 Avg F1 会误判：α=0 看起来最好，但 worst-class 很低，代表类间不均衡；中等 α（例如 25）更均衡。

你现在读完 2/3/4 表，应该形成的“作者实验逻辑”

  1. Table 2（100 类）：先证明 “V2 层级设计 + hYOLO 结构”能带来更好的可靠性（TP 更自信、FP 更不自信）
  2. Table 3（1011 类）：再证明在大规模细粒度下，哪怕 F1 不一定赢，hYOLO 仍然在“置信度校准 + 错分语义接近”上更像一个可部署系统
  3. Table 4（α sweep）：最后解释 loss 里 penalty 的权重为什么是一个“要靠实验选的旋钮”，并展示它如何在“平均性能 vs 最差类”之间做取舍

Ⅴ Conclusion

According to a document from Oct 27, 2025（hYOLO 论文 arXiv v1），第 5 节 Conclusion 主要是在“收束全文贡献 + 强调实验结论 + 给出应用场景 + 提出未来研究方向”。

5.1 作者在 Conclusion 里怎么概括他们做了什么？

作者一句话把贡献打包成 3 件套：

  1. 提出 hYOLO：一个建立在 YOLOv8 之上的端到端层级模型（用于图像检测/分类场景）。
  2. 关键改动包括：层级架构 + 修改过的 loss + 层级标签体系。
  3. 用同一数据集构造了两种 hierarchy（不考虑视觉相似性 vs 考虑视觉相似性）来做对照实验。

5.2 Conclusion 里最核心的实验结论是什么？

作者明确说：当层级结构“显式考虑类之间的视觉相似性”时，模型在检测与分类任务上的表现会显著更好。换句话说，层级不是随便建的；“符合视觉相似性”的层级更能帮助模型学到有效的 coarse-to-fine 决策路径。

这也正好呼应你前面读到的 Table 2 / 3：他们不仅看 F1，还强调 TP/FP 置信度这种“更像部署指标”的改进。

5.3 作者认为 hYOLO 可以用在哪些真实场景？

Conclusion 后半段基本是“应用展望清单”，强调一个共同点：当类别天然存在层级结构时（从粗到细、从大类到子类），hYOLO 会更合适。

论文举例包括（我按他们原文顺序归纳）：

  • 医疗影像：区分视觉相近的疾病亚型（例如不同肿瘤），支持更细致诊断与决策支持。
  • 零售/超市：智能购物车、自助结账、库存管理；层级如 fruit → citrus → orange，能提升遮挡/包装变化/光照变化下的识别鲁棒性。
  • 自动驾驶：如 vehicle → truck → delivery truck 或交通标志层级，提升识别可靠性和上下文决策能力。
  • 安防/公共安全：利用“同语义组内的视觉相似性”降低误报、提升可解释性（论文举了 tool/weapon 的对比例子）。
  • 电商检索/推荐：用层级标签让搜索、推荐、筛选更细粒度、更一致。
  • 生态环境监测：动植物/土地覆盖本来就是 taxonomy，层级预测更自然、更可扩展。
  • 数字艺术/文化遗产：按风格/流派/地域等层级做归档与检索。

5.4 作者给出的未来工作方向（你后面复现实验也会用到）

作者在 Conclusion 结尾列了一串很“研究路线图”的 future work，核心包括：

  • 更深的 hierarchy、替代架构、更强的 loss
  • 制定层级数据集构建的标准化指南（层级怎么定义、类怎么命名、怎么标注）
  • 开发用于评估层级数据集质量的工具与指标
  • 重点解决：层级越深，类不平衡越严重（deep levels 的 imbalance）。

你读完 Conclusion 最该带走的 3 句话

  • hYOLO 的“组合拳”是：层级分支架构 + 层级一致性 loss + 层级标签。
  • 层级必须“像视觉”（考虑视觉相似性）才能显著增益，而不是只做业务/语义树。
  • 真正难点在“模型之外”：怎么建立高质量 hierarchy、怎么评估它、怎么处理深层级不平衡。
